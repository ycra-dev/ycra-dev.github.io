## 핵심 개념

많은 개발자가 "나는 코드만 작성하면 된다"고 생각하지만, 실제로 업무 시간의 대부분은 사람과의 상호작용에 쓰인다. 이메일, 회의, 협업, 요구사항 논의가 모두 사람과의 일이다.

심지어 코드도 주로 사람이 읽기 위해 작성된다. 컴퓨터만을 위한 것이라면 기계어로 직접 작성하면 된다.

## 동작 원리

Dale Carnegie의 "How to Win Friends and Influence People"에서 차용한 핵심 원칙:

1. **모든 사람은 중요하게 느끼고 싶어 한다** - 가장 근본적인 인간의 욕구
2. **비판하지 마라** - 벌보다 보상이 훨씬 효과적이다
3. **상대방이 원하는 것을 먼저 생각하라** - 자기 관점이 아닌 상대방의 관점에서 대화를 시작하라
4. **논쟁을 피하라** - 논쟁에서 이기는 유일한 방법은 논쟁을 피하는 것이다

## 예시

코드 리뷰에서 대화 비교:
```
[나쁜 예시]
"이 코드 설계가 완전히 잘못됐어요. 제가 더 나은 방법을 알려드릴게요."
→ 상대방의 자존심을 깎고, 방어적으로 만듦

[좋은 예시]
"이 접근 방식에 대해 더 이야기해주세요.
혹시 성능 측면에서 다른 방법도 고려해보셨나요?"
→ 상대방을 존중하면서 대안을 제시
```

상사에게 기능 구현 방식을 제안할 때:
```
[비효과적] "저는 이 방식이 더 좋다고 생각합니다"
[효과적]   "이 방식으로 하면 출시 일정을 2주 앞당길 수 있고,
            유지보수 비용도 줄어들 것 같습니다"
```
