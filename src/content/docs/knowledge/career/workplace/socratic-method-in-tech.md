## 핵심 개념

사람은 자신이 발견한 아이디어에 훨씬 더 강한 주인의식을 가진다. 아이디어를 직접 제시하는 것보다 질문을 통해 유도하는 것이 훨씬 높은 동의(buy-in)를 얻는다. 아이디어에 대한 공로를 일부 포기하지만, 아이디어가 실제로 채택될 확률이 높아진다.

## 동작 원리

소크라테스식 유도의 단계:

```
문제 인식 유도:
"이 방식으로 1000개의 동시 요청이 오면 어떻게 될까요?"
↓
해결책 탐색:
"그럼 어떤 방법이 있을까요?"
↓
자신의 해결책 도달:
"스테이징 환경을 만들어서 테스트하면 어떨까요?"
```

교육을 통한 권위 구축: TDD 도입을 원한다면, TDD에 대한 프레젠테이션을 먼저 진행하여 팀을 교육한 후 채택 제안. 프레젠테이션 진행자는 자연스럽게 권위와 신뢰성이 구축됨.

## 예시

- 상사가 "프로덕션에 직접 배포하자"고 할 때: "빨리 배포하는 건 좋은데, 혹시 버그가 고객 데이터를 손상시키면 어떡하죠?" → "테스트 먼저" → "스테이징을 만들면 어떨까요?"
- TDD 도입: 팀에 TDD 프레젠테이션 먼저 → 공감대 형성 → 자연스러운 채택
- 코드 리뷰 설계 문제: "이 경우 1000개의 동시 요청이 오면 어떻게 될까요?" (직접 지적 대신 질문)
