## 핵심 개념

포스트모템의 목적은 비난이 아니라 **집단 지혜 구축**이다. "이 한 시간의 어려운 대화가 다음 한 달간의 팀 프로젝트 성공 확률을 높인다면, 높은 레버리지이며 시간과 감정적 투자의 가치가 충분하다."

프로젝트 수준 포스트모템이 부족한 이유:
- 성공하면 축하하고 바로 다음으로 이동
- 명확한 목표/메트릭 없이 성공 여부 판단 어려움
- 실패를 공개적으로 인정하기 불편
- 새 프로젝트로 바빠서 회고 시간 부족

## 동작 원리

NASA의 사례: 4시간 시뮬레이션 후 1시간 디브리핑. 우주 비행 후 한 달 이상의 종일 디브리핑. 200개 이상의 우주 비행에서 누적된 교훈을 **Flight Rules**로 편찬 — 수천 개의 문제 상황과 해결 단계를 분 단위의 상세함으로 기록.

Toyota의 **Five Whys** 기법:
1. 왜 사이트가 다운됐나? → 서버 과부하
2. 왜 과부하? → 트래픽이 소수 서버에 집중
3. 왜 분산되지 않았나? → 한 고객의 데이터가 해당 서버에만 존재
4. 왜 그렇게 되었나? → 샤딩 전략의 한계
5. 왜 그 전략을 사용? → 초기 설계에서 이 규모를 예상 못 함

증상에서 근본 원인으로 이동하는 것이 핵심이다. Amazon과 Asana는 Five Whys를 사용하며 포스트모템 결과를 널리 공유하여 다른 팀도 동일한 교훈을 학습하도록 한다.

## 예시

```
# 포스트모템 템플릿
post_mortem = {
    "사건 요약": "2월 15일 프로덕션 DB 장애, 2시간 다운타임",
    "타임라인": [
        "14:00 - 알림 발생",
        "14:15 - 온콜 엔지니어 조사 시작",
        "14:45 - 근본 원인 식별 (디스크 풀)",
        "15:30 - 임시 조치 완료",
        "16:00 - 서비스 복구"
    ],
    "근본 원인": "로그 로테이션 스크립트 실패로 디스크 공간 고갈",
    "Five Whys": ["디스크 풀", "로그 미삭제", "cron job 실패",
                  "cron 설정 미검증", "모니터링 부재"],
    "액션 아이템": [
        "디스크 사용량 알림 추가",
        "로그 로테이션 모니터링 추가",
        "월간 인프라 건강 점검 체크리스트 작성"
    ],
    "교훈": "드문 배치 프로세스도 정기 모니터링 필요"
}
```
