## 핵심 개념

머클 트리는 블록체인에서 트랜잭션의 무결성을 검증하고, 분산 시스템에서 데이터 복제본 간의 차이를 효율적으로 탐지하는 데 사용된다.

**머클 트리의 기본 구조**: 이진 머클 트리를 기준으로, 각 데이터 항목의 키 k_i는 해시 함수 h1()으로 해시되어 n비트 해시값을 얻고, 이 값이 리프 노드의 식별자가 된다. 각 데이터 값 v_i는 또 다른 해시 함수 h2()로 해시된다. 리프 노드의 저장 해시값은 해당 리프에 매핑되는 모든 데이터 값의 해시를 h3() 함수로 결합하여 계산한다. 내부 노드의 해시값은 자식 노드들의 해시값을 h3()으로 결합하여 계산한다.

**데이터 무결성 검증**: 두 복제본에 구축된 머클 트리에서, 모든 항목이 동일하면 루트 노드의 해시값도 동일하다. SHA-1과 같이 충분히 긴 해시값을 생성하는 함수를 사용하면, 서로 다른 값이 같은 해시를 가질 확률은 극히 낮다. 따라서 루트 해시값이 같으면 두 복제본의 데이터가 동일하다고 확신할 수 있다.

**차이 탐지 알고리즘**: 루트 해시값이 다르면, 각 자식의 해시값을 비교하고, 차이가 있는 자식 방향으로 탐색을 진행한다. 리프에 도달하면 데이터 항목을 직접 비교한다. 한 번의 탐색은 리프 수의 로그에 비례하는 시간이 걸린다. 전체 비용은 O(m log_2 N)으로, m은 차이가 있는 항목 수, N은 총 항목 수이다.

**블록체인에서의 활용**: Bitcoin을 포함한 많은 블록체인은 블록 내의 트랜잭션을 머클 트리에 저장한다. 이를 통해 악의적 사용자에 의한 데이터 손상을 효율적으로 감지할 수 있다. 사용된(spent) 트랜잭션으로만 구성된 서브트리를 잘라내어 저장 공간을 줄이는 최적화도 가능하다.

**라이트 노드와의 관계**: 전체 블록체인을 저장하지 않는 라이트 노드(light node)는 트리의 루트 해시만 보유하면 된다. 풀 노드(full node)가 필요한 데이터와 검증에 필요한 해시를 제공하면, 라이트 노드는 자신이 보유한 루트 해시와 비교하여 데이터의 일관성을 확인할 수 있다.

**머클-패트리시아 트리(Merkle-Patricia Tree)**: Ethereum 등 상태를 유지하는 블록체인에서 사용되는 변형으로, 키 기반 검색을 효율적으로 지원한다. 노드를 직접 삭제/삽입하지 않고 새 루트를 생성하며, 이전 트리의 서브트리를 참조(재사용)한다. 이전 트리는 불변(immutable)이므로 블록체인의 불변성 속성과 잘 부합한다.

## 예시

8개 데이터 항목에 대한 머클 트리 구조:

```
                    루트
                 h3(v0, v1)
                /          \
          h3(v00, v01)    h3(v10, v11)
           /      \        /       \
        v00      v01    v10       v11
       /           |      \         \
   h2(i1,i4)  h2(i2)  h2(i5,i7)  h2(i3,i6)

리프 노드: 데이터 항목들의 해시 조합
내부 노드: 자식 해시들의 조합
루트 노드: 전체 데이터의 해시 요약
```

머클 트리를 이용한 차이 탐지:

```
복제본 A의 머클 트리:
    루트 해시 = ABC123     ← 비교 시작

복제본 B의 머클 트리:
    루트 해시 = DEF456     ← 불일치!

1. 루트 불일치 → 자식 비교
2. 왼쪽 자식 일치, 오른쪽 자식 불일치
3. 오른쪽 자식의 자식 비교
4. 리프 도달 → 구체적 차이 항목 식별

비용: O(m * log2(N))
  m = 차이 항목 수
  N = 총 항목 수
```
