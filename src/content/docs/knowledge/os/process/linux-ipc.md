## 핵심 개념

Linux IPC는 **시그널, 파이프, 공유 메모리** 등 다양한 메커니즘을 통해 프로세스 간 이벤트 알림과 데이터 전달을 가능하게 한다. UNIX 철학에서 작은 프로그램들을 조합하여 복잡한 작업을 수행하려면 프로세스 간 데이터 교환이 필수적이다.

통신 유형(이벤트 알림 vs 데이터 전송)과 프로세스 관계(부모-자식 vs 무관한 프로세스)에 따라 적합한 메커니즘이 다르다.

## 동작 원리

### IPC 메커니즘 비교

| 메커니즘 | 데이터 전송 | 동기화 | 관계 제약 | 성능 |
|----------|-----------|--------|----------|------|
| **시그널** | 이벤트만 (정보 X) | 비동기 | 없음 | 매우 빠름 |
| **파이프** | 바이트 스트림 | 동기화 내장 | 부모-자식 | 빠름 |
| **명명된 파이프** | 바이트 스트림 | 동기화 내장 | 없음 | 빠름 |
| **공유 메모리** | 임의 데이터 | 별도 필요 | 없음 | 가장 빠름 |
| **세마포어** | 동기화만 | 동기화 자체 | 없음 | 빠름 |
| **메시지 큐** | 구조화된 메시지 | 동기화 내장 | 없음 | 중간 |
| **소켓** | 바이트 스트림 | 동기화 내장 | 없음 (네트워크 포함) | 가장 유연 |

### 1. 시그널 (Signal)

```
송신자 → kill(pid, SIGTERM) → 커널 → 수신자의 시그널 핸들러 실행
```

- 제한된 수의 시그널 (SIGKILL, SIGTERM, SIGUSR1 등)
- **정보 전달 불가** (이벤트 발생 여부만)
- 커널도 시그널 생성 (SIGSEGV, SIGCHLD 등)

비유하면 비상벨과 같다. 무슨 일인지는 모르지만 주목하라는 알림.

### 2. 파이프 (Pipe)

```
[익명 파이프]
부모: pipe(fd) → fork()
      fd[1] (쓰기)  fd[0] (읽기)
         ↓              ↓
자식:   fd[0] (읽기) ← 파이프 버퍼 ← fd[1] (쓰기)

$ ls | grep txt   ← 셸이 파이프로 연결
```

비유하면 수도관과 같다. 한 쪽에서 넣으면 다른 쪽에서 나온다.

### 3. 공유 메모리 (Shared Memory)

```
프로세스 A 주소공간    프로세스 B 주소공간
       ↓                    ↓
    VMA A  ──→  물리 페이지  ←──  VMA B
                (공유 메모리 객체)
```

- 데이터 복사 없이 직접 접근 → **가장 빠름**
- 동기화는 별도로 필요 (세마포어, mutex 등)
- 프로세스 종료 후에도 객체 유지 가능 (persistent)

비유하면 공유 화이트보드와 같다. 모두가 직접 읽고 쓸 수 있지만, 동시에 쓰면 충돌한다.

## 예시

- **시그널**: 터미널에서 `Ctrl+C` → 셸이 SIGINT를 포그라운드 프로세스에 전달
- **파이프**: `cat file.txt | wc -l` → cat의 stdout이 wc의 stdin에 연결
- **공유 메모리**: 데이터베이스 프로세스들이 버퍼 캐시를 공유
