## 핵심 개념

시그널(Signal)은 프로세스에게 비동기적으로 이벤트 발생을 알리는 소프트웨어 인터럽트 메커니즘입니다. 프로세스가 예외 상황(키보드 인터럽트, 오류, 타이머)에 즉시 대응해야 하는데, 폴링 방식은 비효율적이기 때문에 비동기 이벤트 처리와 프로세스 간 간단한 통신 수단으로 사용됩니다.

## 동작 원리

각 시그널에는 고유 번호가 있고, 프로세스는 시그널에 대해 기본 동작, 무시, 또는 핸들러 지정이 가능합니다.

1. **시그널 발생**: 키보드(`^C` → SIGINT), 오류(SIGSEGV), `kill()` 시스템 콜, 타이머 등
2. **시그널 전달**: 커널이 대상 프로세스에 시그널을 pending 상태로 기록
3. **시그널 처리**: 프로세스가 다음 실행될 때 처리
   - 기본 동작 (종료, 코어 덤프 등)
   - 무시 (ignore)
   - 사용자 정의 핸들러 호출

```
시그널 발생원                    프로세스
    │                              │
    │──── kill(pid, SIGINT) ──────>│
    │                              │
                              [핸들러 실행]
                              또는 [기본 동작]
```

### 주요 시그널

| 시그널 | 번호 | 트리거 | 기본 동작 |
|--------|------|--------|----------|
| SIGINT | 2 | `^C` | 종료 |
| SIGQUIT | 3 | `^\` | 종료 + 코어 덤프 |
| SIGKILL | 9 | `kill -9` | 종료 (무시 불가) |
| SIGSEGV | 11 | 잘못된 메모리 접근 | 종료 + 코어 덤프 |
| SIGCONT | 18 | `fg`/`bg` | 재개 |
| SIGSTOP | 19 | - | 정지 (무시 불가) |

## 예시

시그널은 "문자 메시지"와 같습니다. 보내는 측은 언제든 보낼 수 있고, 받는 측은 자기 상황에 맞춰 처리합니다.

- 간단하고 빠른 비동기 통신 가능
- 하드웨어 인터럽트와 유사한 개념으로 직관적
- 프로세스 제어(정지, 재개, 종료)에 유용
- 단점: 동일 시그널 중복 발생 시 손실 가능(큐잉 없음), 전달 가능한 정보가 제한적(시그널 번호뿐)
- 핸들러 내에서 호출 가능한 함수 제한(async-signal-safe)
- 4.2BSD에서 신뢰성 있는 시그널로 개선
