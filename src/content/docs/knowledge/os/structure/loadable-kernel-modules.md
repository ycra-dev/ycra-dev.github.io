## 핵심 개념

커널이 **핵심 구성 요소**를 가지고, **부팅 시 또는 런타임**에 모듈을 통해 추가 서비스를 **동적으로 링크**하는 운영체제 설계 방식이다. 현재 운영체제 설계의 가장 좋은 방법론으로 평가된다.

## LKM 설계 철학

- 커널은 핵심 서비스 제공
- 다른 서비스는 커널 실행 중 동적으로 구현
- 서비스를 커널에 직접 링크하는 것보다 효율적

## 사용 운영체제

- Linux
- macOS
- Solaris
- Windows

## Linux에서의 LKM

주로 **장치 드라이버**와 **파일 시스템 지원**에 사용된다:

- 시스템 시작(부팅) 시 또는 런타임에 "삽입" 가능
- 예: USB 장치를 실행 중인 시스템에 연결 시, 필요한 드라이버가 없으면 동적 로드
- 런타임에 커널에서 제거도 가능

## 다른 접근법과의 비교

| 비교 대상 | 유사점 | 차이점 |
|-----------|--------|--------|
| **계층적 시스템** | 각 커널 섹션이 정의된 보호 인터페이스를 가짐 | 더 유연함: 어떤 모듈이든 다른 모듈 호출 가능 |
| **마이크로커널** | 주 모듈이 핵심 기능만 가지고 다른 모듈과 통신 방법을 앎 | 더 효율적: 메시지 패싱 불필요 |

## LKM의 이점

- 동적이고 모듈식 커널 허용
- 모놀리식 시스템의 성능 이점 유지
- 예: CPU 스케줄링과 메모리 관리 알고리즘을 커널에 직접 컴파일하고, 다양한 파일 시스템은 모듈로 추가

## 트레이드오프

| 장점 | 단점 |
|------|------|
| 런타임에 기능 추가/제거 가능 | 모듈 간 버전 호환성 문제 가능 |
| 마이크로커널과 달리 메시지 패싱 불필요 | 보안 취약점이 될 수 있음 |
| 커널 크기 최소화 (필요한 모듈만 로드) | |
| 전체 커널 재컴파일 없이 모듈 개발/테스트 가능 | |
