## 핵심 개념

디지털 서명(Digital Signature)은 발신자가 자신의 개인키로 메시지(또는 메시지의 해시)를 암호화하여, 수신자가 공개키로 검증함으로써 발신자의 신원과 메시지의 무결성을 보장하는 기술이다.

## 동작 원리

디지털 서명은 공개키 암호를 **역방향으로** 사용한 것이다. 일반적으로 공개키로 암호화하고 개인키로 복호화하는 반면, 디지털 서명은 개인키로 암호화하고 공개키로 복호화한다.

**보장하는 세 가지 속성**:
1. **인증(Authentication)**: 서명은 개인키 소유자만 생성할 수 있으므로, 발신자의 신원이 확인된다.
2. **무결성(Integrity)**: 서명된 후 메시지가 변경되면 검증이 실패한다.
3. **부인 방지(Non-repudiation)**: 발신자는 나중에 자신이 메시지를 보내지 않았다고 부인할 수 없다.

**실제 서명 과정** (메시지 다이제스트와 조합):
메시지 전체를 개인키로 암호화하면 너무 느리므로, 실제로는 메시지의 **해시(다이제스트)**를 계산하고, 이 해시를 개인키로 암호화한다.

1. 발신자: 메시지의 해시 계산 → 개인키로 해시를 암호화 = 디지털 서명
2. 발신자: 원본 메시지 + 디지털 서명을 함께 전송
3. 수신자: 받은 메시지의 해시를 직접 계산
4. 수신자: 디지털 서명을 발신자의 공개키로 복호화 → 원래 해시 복원
5. 수신자: 두 해시가 일치하면 → 메시지가 변조되지 않았고, 발신자가 확인됨

**인증서(Certificate)**:
공개키가 정말로 주장하는 사람의 것인지 확인하는 문제가 남는다. 이를 해결하기 위해 신뢰할 수 있는 인증 기관(CA)이 공개키와 소유자 정보를 디지털 서명한 인증서를 발급한다. HTTPS에서 브라우저가 확인하는 것이 바로 이 인증서이다.

## 예시

디지털 서명 과정:
```
발신자 (Alice):
1. 메시지: "100만원을 Bob에게 이체합니다"
2. 해시 계산: SHA-256("100만원을...") = a7f3b2c1...
3. 서명 생성: RSA_encrypt(a7f3b2c1..., Alice의 개인키) = 서명값
4. 전송: {메시지 + 서명값}

수신자 (은행):
1. 메시지에서 해시 계산: SHA-256("100만원을...") = a7f3b2c1...
2. 서명 검증: RSA_decrypt(서명값, Alice의 공개키) = a7f3b2c1...
3. 두 해시 비교: a7f3b2c1... == a7f3b2c1... ✓ → 검증 성공

만약 메시지가 "1000만원을..."으로 변조되었다면:
1. 해시: SHA-256("1000만원을...") = 5d8e9f4a... (다른 값)
2. 서명에서 복원된 해시: a7f3b2c1...
3. 불일치! → 변조 감지
```

HTTPS 인증서 확인:
```
1. 브라우저가 amazon.com에 접속
2. 서버가 인증서 제공: "이 공개키는 amazon.com의 것입니다 - VeriSign 서명"
3. 브라우저에 내장된 VeriSign의 공개키로 서명 검증
4. 검증 성공 → 주소창에 자물쇠 아이콘 표시
5. 검증 실패 → "이 사이트는 안전하지 않습니다" 경고
```
