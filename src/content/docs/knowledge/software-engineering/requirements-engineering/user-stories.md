## 핵심 개념

사용자 스토리는 "As a [역할], I want [기능] so that [이유]" 형식으로 작성되며, 요구사항을 자연어로 간결하게 표현한다. 각 스토리는 개발 팀이 하나의 반복(iteration) 내에서 완료할 수 있을 정도로 작아야 한다. 사용자 스토리는 대화의 출발점으로, 세부 사항은 개발자와 고객 간의 대화를 통해 명확해진다. 스토리는 스토리 포인트로 크기가 추정되며, 속도(velocity)를 통해 팀의 생산성을 측정한다.

## 예시

사용자 스토리 예시: "처방의사로서, 처방을 내릴 때 환자의 알레르기 정보를 보여주는 기능이 필요하다. 이는 알레르기 반응을 유발하는 약물 처방을 방지하기 위해서이다."
