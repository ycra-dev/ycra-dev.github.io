## 핵심 개념

문제 소유(Problem Ownership)는 팀의 정체성을 특정 제품(솔루션)이 아닌 해결해야 할 문제에 고정하는 조직 설계 원칙이다. 제품은 솔루션에 불과하며 대체될 수 있지만, 잘 선택된 문제는 영원히 유효하다.

## 동작 원리

팀을 특정 제품에 묶으면 위험한 결과가 발생한다. 팀이 그 솔루션을 자신의 정체성과 자존감의 일부로 여기게 되어, 더 나은 대안이 나타나도 기존 솔루션을 방어하고 변화에 저항한다.

| 제품 소유 | 문제 소유 |
|-----------|-----------|
| "Git 저장소를 관리하는 팀" | "회사에 버전 관리를 제공하는 팀" |
| 새 기술로 전환 시 저항 | 새 기술 자유롭게 실험 가능 |
| 솔루션 보호에 집중 | 문제 해결에 집중 |

이 원칙은 여러 수준에서 적용된다:
- 팀 수준: 특정 기술 스택이 아닌 비즈니스 문제를 소유
- 조직 수준: 특정 제품이 아닌 고객 문제를 소유
- 개인 수준: 특정 도구의 전문가가 아닌 문제 영역의 전문가로 성장

이 접근법은 소프트웨어의 지속가능성(sustainability)과 직결된다. 기술은 변하지만 문제는 지속되기 때문이다.

## 예시

Google 검색 지연시간 문제를 다루는 팀들은 "latency 최적화 도구 팀"이 아니라 "검색 지연시간 문제를 해결하는 팀"으로 정의되었다. 이 덕분에 코드 최적화, 메트릭 개선, 개발자 교육, 정책 변경 등 다양한 접근을 자유롭게 탐색할 수 있었다. 솔루션에 묶여있었다면 코드 최적화만 반복했을 것이다.
