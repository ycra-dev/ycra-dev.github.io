## 배운 것

### 네트워크 OS vs 분산 OS

- **NOS (Network OS)**: 사용자가 원격 자원에 **명시적으로** 접근 (SSH, FTP 등)
  - 현재 모든 범용 OS(Windows, macOS, Linux)가 NOS
- **DOS (Distributed OS)**: 원격 자원을 **로컬처럼 투명하게** 접근
  - 마이그레이션 유형: 데이터 / 계산(RPC) / 프로세스

### 분산 시스템의 핵심 특성

**투명성 (Transparency)**
- 위치 투명성: 파일 이름에서 물리적 위치가 드러나지 않음
- 위치 독립성: 파일이 이동해도 이름 유지 (더 강한 속성)
- 복제/마이그레이션/동시성/실패 투명성 등

**견고성 (Robustness)**
- 장애 유형: 네트워크 파티션, 호스트 장애
- 해결책: 복제(Replication), 중복 제거, 상태 복구
- 복구 가능성(Recoverability): 영구/휘발/안정 스토리지 구분

**확장성 (Scalability)**
- 수평 확장(Scale-out): 머신 추가로 처리 능력 증가
- 수직 확장(Scale-up): 기존 머신 성능 향상
- 트레이드오프: 중앙 집중 vs 분산 조정, 강한 일관성 vs 약한 일관성

### 분산 파일 시스템 (DFS)

**아키텍처 모델**
1. **클라이언트-서버 모델** (NFS, OpenAFS)
2. **클러스터 기반 모델** (GFS, HDFS)

**NFS vs OpenAFS**

| 특성 | NFS | OpenAFS |
|------|-----|---------|
| 설계 목표 | 빠른 장애 복구 | 확장성 |
| 서버 상태 | 무상태 (Stateless) | 상태 유지 (콜백) |
| 캐싱 단위 | 블록 | 전체 파일 |
| 특징 | Chatty 프로토콜 | 전역 네임스페이스 |

**클러스터 기반 DFS (GFS, HDFS)**
- 파일을 64MB~128MB 청크로 분할
- 각 청크를 3개 이상 복제하여 장애 대비
- 메타데이터 서버 + 데이터 서버 분리 구조
- "하드웨어 장애는 일상"이라는 전제로 설계

**DFS 캐싱 정책**
- Write-Through: 즉시 서버 반영 (신뢰성 높음)
- Write-on-Close: 파일 닫을 때 전송 (OpenAFS 방식)
- 일관성 관리: 클라이언트 주도 vs 서버 주도(콜백)
